## 🎯 Цели проекта

**Домкрат** — это многосторонний маркетплейс автотоваров и запчастей для рынка Узбекистана со следующими ключевыми характеристиками:

- **B2C** покупки автотоваров обычными клиентами
- **B2B2C** модель: платформа агрегирует мерчантов и продаёт их товары
- **Полноценный WMS** (Warehouse Management System) для склада платформы
- **Локализация**: русский и узбекский языки (с переключателем)
- **Локальные платёжные системы**: Click, Payme, Uzum + COD (наличные)
- **Гибридная монетизация**: комиссия + аренда склада + подписка мерчантов

---

## 👥 Целевая аудитория

### Покупатели (B2C)
- Владельцы личных автомобилей
- Автосервисы (мелкооптовые покупки)
- Корпоративные клиенты (юр.лица с оплатой по счёту)

### Мерчанты (продавцы)
- **Тип 1 (FBO — Fulfillment by Operator)**: привозят товар на склад платформы
- **Тип 2 (FBS — Fulfillment by Seller)**: хранят товар у себя, отгружают сами или вызывают курьера платформы

## 🌐 Структура продукта

Проект состоит из **5 основных частей**:

### 1. **Каталог (клиентская часть)** — `domkrat.uz`
- Публичный сайт-каталог
- Поиск по марке/модели/году авто
- Поиск по VIN-коду
- Поиск по артикулу/OEM
- Корзина, оформление заказа, оплата
- Отзывы и рейтинги

### 2. **Личный кабинет покупателя** — `domkrat.uz/account`
- История заказов, статусы
- Гаражи (сохранённые авто пользователя)
- Избранное, списки
- Адреса доставки
- Возвраты

### 3. **Кабинет мерчанта** — `merchant.domkrat.uz`
- Управление товарами и остатками
- Заказы на обработку (для Тип 2)
- Финансы: продажи, комиссии, аренда, запрос на вывод
- Аналитика и отчёты
- Документы (накладные, акты)

### 4. **Админ-панель** — `admin.domkrat.uz`
- Управление пользователями и ролями
- Модерация товаров и мерчантов
- Управление каталогом авто (марки/модели/модификации)
- Управление складом (зоны, стеллажи, ячейки)
- Заказы (все), споры, возвраты
- Финансы платформы, ручной вывод мерчантам
- Контент (баннеры, страницы, SEO)

### 5. **Мобильное приложение для склада/курьеров** (PWA/Native)
- Складские операции (приёмка, размещение, отбор по QR/штрих-коду)
- Маршруты курьеров
- Подтверждение выдачи заказа

## ⚙️ Ключевые особенности проекта

### Каталог автотоваров
- **Иерархическая категоризация**: Категория → Подкатегория → Тип товара
- **Совместимость с авто**: гибридная (марка/модель/год + VIN-поиск через API)
- **Применимость**: один товар может подходить к разным авто (many-to-many)
- **Кросс-номера (OEM)**: связь оригинальных номеров и аналогов

### Складская логистика (WMS)
- **Полная иерархия мест**: Склад → Зона → Стеллаж → Полка → Ячейка
- **FIFO** для ротации товаров
- **Партионный учёт** (опционально на будущее)
- **Контроль неликвида**: алерт если товар лежит >3 месяцев
- **Поддержка мультискладов** (на будущее)

### Финансы
- **Гибридная монетизация**:
  - Комиссия с продаж (% от стоимости товара)
  - Аренда складских мест для Тип 1 мерчантов
  - Подписка для расширенных возможностей мерчантов
- **Ручной вывод средств админом** по запросу мерчанта
- **Раздельный учёт по мерчантам** (балансы, истории операций)

### Доставка
- **4 типа доставки**:
  1. Самовывоз со склада/пунктов выдачи
  2. Курьер платформы (по Ташкенту)
  3. Доставка по Узбекистану (BTS, Yandex Go — интеграции)
  4. Доставка силами мерчанта (Тип 2)

### Платежи
- **Click, Payme, Uzum** — локальные платёжные системы
- **COD** — наличные при получении
- **Банковский перевод** для юр.лиц (на будущее)

### Локализация
- **2 языка**: русский (default) и узбекский
- Все пользовательские данные (названия товаров, описания, категории) хранятся в БД с переводами
- Переключатель языка в шапке сайта

## 🎬 Высокоуровневый флоу заказа

```
┌─────────────────────────────────────────────────────────────────┐
│ 1. Клиент ищет товар (по марке/модели/VIN/артикулу)             │
│ 2. Добавляет в корзину                                          │
│ 3. Оформляет заказ (адрес, способ доставки, оплата)             │
│ 4. Оплачивает (Click/Payme/Uzum/COD)                            │
│ 5. Заказ распределяется по мерчантам:                           │
│    - Тип 1 → склад платформы получает задачу на сборку          │
│    - Тип 2 → задача уходит мерчанту в его кабинет               │
│ 6. Складской работник собирает заказ (по локациям с QR)         │
│ 7. Передача курьеру / в пункт выдачи                            │
│ 8. Клиент получает заказ, подтверждает / отказывается           │
│ 9. Финансовое закрытие: списание с баланса, начисление комиссии │
└─────────────────────────────────────────────────────────────────┘
```

## 🚀 Этапы разработки

### Phase 1 — MVP (3-4 месяца)
- Базовый каталог с поиском
- Регистрация и кабинеты (покупатель, мерчант, админ)
- Заказы и оплата (Click/Payme/Uzum + COD)
- Упрощённый склад (ручной ввод остатков)
- Базовая доставка (самовывоз + курьер по Ташкенту)
- Локализация RU/UZ

### Phase 2 — Расширение (2-3 месяца)
- Полноценный WMS с QR-кодами и мобильным приложением
- Партионный учёт и FIFO
- Алерты по неликвиду
- VIN-поиск через сторонний API
- Интеграция с Yandex Go, BTS
- Финансовая аналитика для мерчантов

### Phase 3 — Масштабирование (3+ месяца)
- Мультискладская модель
- B2B-кабинет (опт, договоры)
- Программа лояльности
- Мобильное приложение покупателя
- Расширенная аналитика, BI
- Партнёрская / реферальная система

## ✅ Принципы разработки

1. **Modular monolith** на старте — лёгкое разделение на микросервисы потом
2. **Domain-Driven Design** — чёткие границы модулей
3. **Event-driven** для асинхронных процессов (через RabbitMQ/Redis)
4. **API-first** — все интерфейсы через REST/GraphQL
5. **TypeScript everywhere** — типизация на бэке и фронте
6. **Test coverage** — минимум 70% для critical paths
7. **Observability** — логи (Pino), метрики (Prometheus), трейсинг (OpenTelemetry)
8. **Security by default** — JWT, RBAC, rate limiting, CSRF protection
