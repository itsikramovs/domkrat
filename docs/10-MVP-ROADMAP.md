## Phase 1: MVP (12-16 недель)

### Цель
Минимально жизнеспособный маркетплейс, который:
- Принимает реальные заказы
- Обрабатывает онлайн-платежи
- Имеет верифицированных мерчантов
- Доставляет товары по Ташкенту

### Sprint Plan (по 2 недели)

#### Sprint 1-2: Foundation (Недели 1-4)

**Цели**: Подготовить инфраструктуру, базовые модели

**Задачи**:
- [ ] Настройка монорепо (pnpm workspaces)
- [ ] Базовая структура NestJS API
- [ ] Базовая структура Next.js приложений (web, merchant, admin)
- [ ] PostgreSQL + Prisma — все таблицы из схемы
- [ ] Docker Compose для локальной разработки
- [ ] GitHub Actions CI: lint + tests
- [ ] Auth модуль: регистрация, вход, JWT
- [ ] SMS-интеграция с Eskiz (mock + sandbox)
- [ ] Базовая UI-система (shadcn/ui + Tailwind)
- [ ] Дизайн-токены, бренд-стиль «Домкрат»
- [ ] Sentry для ошибок
- [ ] Логирование (Pino)

**Демо**: пользователь может зарегистрироваться через SMS, войти, увидеть пустой профиль.

---

#### Sprint 3-4: Каталог (Недели 5-8)

**Цели**: Полноценный каталог товаров

**Задачи**:
- [ ] Категории (дерево) — admin + public API
- [ ] Бренды
- [ ] Атрибуты и характеристики
- [ ] Товары — CRUD для админа
- [ ] Загрузка изображений в MinIO + ресайз через Sharp
- [ ] Справочник авто: марки, модели, поколения, модификации
- [ ] Совместимость товар↔авто (M:N)
- [ ] Frontend: страница категории, карточка товара
- [ ] Базовый поиск через PostgreSQL FTS (Meilisearch позже)
- [ ] Сидинг тестовых данных: 50 категорий, 500 товаров
- [ ] SEO: meta-теги, sitemap.xml, robots.txt
- [ ] Мультиязычность (RU/UZ) на фронте и API
- [ ] Карусели на главной (новинки, хиты)

**Демо**: посетитель может найти товар, посмотреть детали, фильтровать по марке/модели.

---

#### Sprint 5-6: Корзина и заказы (Недели 9-12)

**Цели**: От корзины до оплаченного заказа

**Задачи**:
- [ ] Корзина (Redis для гостей, БД для авториз.)
- [ ] Промокоды
- [ ] Расчёт стоимости (товары + доставка + НДС)
- [ ] Адреса доставки в кабинете
- [ ] Гараж пользователя
- [ ] Checkout flow: выбор адреса, способа доставки, оплаты
- [ ] Заказ — создание, split на sub_orders
- [ ] Резервирование товара через Redis lock
- [ ] State machine заказа
- [ ] Интеграция Click (полностью + webhooks)
- [ ] Интеграция Payme (полностью + webhooks)
- [ ] COD (Cash on Delivery)
- [ ] Email-уведомления (Mailgun)
- [ ] SMS-уведомления о статусе заказа
- [ ] История заказов в кабинете покупателя
- [ ] Tracking страница (статус заказа)

**Демо**: покупатель проходит полный путь — каталог → корзина → checkout → оплата → подтверждение.

---

#### Sprint 7-8: Кабинет мерчанта (Недели 13-16)

**Цели**: Мерчанты могут управлять своими товарами

**Задачи**:
- [ ] Регистрация мерчанта (форма + загрузка документов)
- [ ] Админ-проверка мерчантов
- [ ] Кабинет мерчанта: дашборд
- [ ] Управление товарами (CRUD)
- [ ] Загрузка фото
- [ ] Управление совместимостью с авто
- [ ] Просмотр своих заказов (sub_orders)
- [ ] Подтверждение заказа (Type 2)
- [ ] Управление остатками (Type 2)
- [ ] Финансовый кабинет: баланс, история, комиссия
- [ ] Запрос на вывод средств
- [ ] Базовая аналитика: продажи, топ товаров
- [ ] Сотрудники мерчанта (под-аккаунты с permissions)

**Демо**: мерчант регистрируется, заводит товары, получает заказ, обрабатывает его.

## Риски и митигация

### Технические риски

| Риск | Влияние | Вероятность | Митигация |
|------|---------|-------------|-----------|
| Сложность интеграции с локальными платёжными системами | Высокое | Среднее | Раннее начало интеграции (Sprint 5), sandbox-тесты |
| Производительность поиска при росте каталога | Среднее | Высокое | Meilisearch с самого начала (Phase 2), правильные индексы PostgreSQL |
| Race conditions при резервации товара | Высокое | Среднее | Redis-locks + транзакции БД |
| Безопасность платежей | Критичное | Низкое | Никогда не храним карты, проверка подписей webhooks |
| Утечка данных мерчантов | Высокое | Низкое | RLS на уровне БД, CASL на уровне API, аудит |

### Бизнес-риски

| Риск | Влияние | Митигация |
|------|---------|-----------|
| Малое количество мерчантов на старте | Высокое | Активная работа с продажами, выгодные условия для первых 20 мерчантов |
| Низкое качество карточек товаров | Среднее | Команда контент-менеджеров, helping мерчантам с фото |
| Слабая узнаваемость бренда | Среднее | SMM, SEO, партнёрства с автосервисами |
| Сложность логистики по Узбекистану | Высокое | Начать с Ташкента, расширяться поэтапно |
| Конкуренция с существующими (uybor, olx) | Среднее | Специализация — мы только запчасти, лучший UX подбора по авто |

### Регуляторные риски

| Риск | Митигация |
|------|-----------|
| Изменения в законодательстве UZ по e-commerce | Юрист в команде, регулярный мониторинг |
| Требования по локализации данных | Все данные хранятся в UZ (через локальные провайдеры) |
| Налоговое регулирование маркетплейсов | Чёткие договоры с мерчантами, корректный учёт НДС |

## Принципы разработки на всех этапах

1. **Iterate fast, but don't break things** — деплоить часто, но через тесты
2. **API-first** — даже фронт сначала ходит в API
3. **Data integrity > performance** — лучше медленнее, но без потери данных
4. **Documentation as code** — Swagger автогенерация
5. **Observability everywhere** — каждый сервис должен быть наблюдаем
6. **Security by default** — каждая фича проходит security review
7. **Mobile-first design** — большинство пользователей зайдут с телефона
8. **Localization first** — все тексты сразу на двух языках
9. **No tech debt accumulation** — рефакторинг в каждом спринте
10. **Customer feedback loop** — каждый sprint демо с реальными мерчантами/покупателями

## Поэтапный rollout

### Pre-launch (последние 4 недели MVP)
1. **Soft launch**: 5-10 пилотных мерчантов + 50-100 тестовых покупателей
2. **Bug-bash сессии**
3. **Нагрузочные тесты**: симуляция 100 concurrent users
4. **Security audit** внешним аудитором

### Launch
1. Запуск с 20-30 верифицированными мерчантами
2. Маркетинговая кампания (соцсети, контекстная реклама)
3. Партнёрство с 1-2 автосервисами для рекомендаций
4. Программа "Первый заказ — скидка 10%"

### Post-launch (первые 3 месяца)
1. Еженедельные ретроспективы
2. Ежедневный мониторинг ключевых метрик
3. Hot-fixes по критичным багам в течение 4 часов
4. Major releases — раз в 2 недели
5. Открытие набора новых мерчантов после стабилизации
