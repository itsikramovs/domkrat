## Роли в AI-команде

### 🧑‍💼 Tech Lead (человек) — критичная роль

**Ответственности**:
- Архитектурные решения (что-как должно быть устроено)
- Декомпозиция фичей на задачи для Claude
- Code review всех PR
- Разрешение тупиков и неоднозначностей
- Прямой контакт с бизнесом

**Скиллы**:
- Глубокая экспертиза в NestJS/TypeScript/PostgreSQL
- Понимание domain-driven design
- Умение читать код быстро
- Опыт работы с AI-инструментами

**Загрузка**: 100%, full-time

---

### 🧑‍💻 Senior Developer (человек) — помощник Tech Lead

**Ответственности**:
- Code review (50% PRs)
- Сложные задачи где Claude не справляется
- Production deployments
- Debugging сложных проблем
- Парное программирование с Claude (особо сложные задачи)

**Загрузка**: 100%

---

### 🤖 Claude Code (AI) — основной разработчик

**Ответственности**:
- 80% написания кода
- Все тесты
- Документация
- Рутинный рефакторинг
- Багфиксы средней сложности

**"Загрузка"**: ~24/7, но требует human-in-the-loop

---

### 🧑‍💼 Product Manager (человек)

**Ответственности**:
- Бизнес-требования
- Приоритезация
- Связь с заказчиком
- User stories
- UAT тесты

**Загрузка**: 100%

---

### 🎨 Designer (человек)

**Ответственности**:
- UI/UX дизайн в Figma
- Дизайн-токены
- Прототипы

**Загрузка**: 50-100% (первые 2 месяца full-time)

---

### 🧪 QA (опционально)

**Ответственности**:
- Manual testing
- Тест-планы
- UAT

**Загрузка**: 50% (можно совместить с PM на старте)

---

### 🛠️ DevOps (опционально)

**Ответственности**:
- Production-серверы
- Мониторинг
- Backups

**Загрузка**: 25-50% (можно contractor)

## Минимальный состав человеческой команды

### Для MVP (3-4 месяца)

**Минимум**:
- 1× Tech Lead (full-time)
- 1× Senior Developer (full-time)
- 1× PM (full-time)
- 1× Designer (full-time first 2 months)
- 1× DevOps contractor (по часам)

**= 3-4 человека + Claude Code**

Для сравнения, традиционная команда на такой проект:
- 1 Tech Lead
- 4-5 Backend разработчиков
- 3-4 Frontend разработчиков
- 2 QA
- 1 DevOps
- 1 Designer
- 1 PM

**= 12-15 человек**

**Экономия**: 3-5x на FTE при правильной организации.

## Daily workflow

### Утренний стендап (15 мин)

**Участники**: Tech Lead, Senior Dev, PM (без Claude)

**Повестка**:
1. Что вчера сделали (включая работу с Claude)
2. Что сегодня планируем
3. Blockers
4. Какие PR на review

---

### Типичный день Tech Lead

```
09:00 — Стендап
09:15 — Review PR от Claude за ночь / утро (1-2 шт)
10:00 — Создание новых задач:
        - Анализ требований
        - Декомпозиция
        - Написание промптов
11:00 — Запуск Claude на новые задачи
        (он работает 1-3 часа на задачу)
11:00 — Параллельно: архитектурная работа
        - Документация
        - ADR (Architecture Decision Records)
13:00 — Обед
14:00 — Review PR от Claude
        - Если нужны правки → комментарии в PR
        - Claude вносит правки за 15-30 мин
14:30 — Merge готовых PR
15:00 — Парное программирование с Claude
        на особо сложной задаче
17:00 — Review результатов дня, planning на завтра
17:30 — Деплой на staging если есть готовый код
18:00 — End of day
```

### Типичный день Senior Dev

```
09:00 — Стендап
09:15 — Review PR от Claude (2-3 шт)
11:00 — Работа над сложной задачей лично
        (где Claude не справится)
13:00 — Обед
14:00 — Подбор и настройка локального окружения
        для тестирования PR от Claude
15:00 — Manual QA готовых фичей
16:00 — Помощь Tech Lead с review
17:00 — Дебаг production issues (если есть)
```

## Управление задачами

### Структура задачи в трекере (Linear / Jira)

```markdown
# DKR-123: Создать модуль Products

## Тип
Feature

## Эстимейт
Claude: 2-3 часа
Review: 30 мин
Total: ~3 часа

## Описание (для бизнеса)
Мерчанты должны иметь возможность управлять своими товарами через кабинет.

## Acceptance criteria (для проверки)
- [ ] Мерчант может создать товар
- [ ] Мерчант видит только свои товары
- [ ] Можно загружать фото (до 10 шт)
- [ ] Поддерживается мультиязычность (RU/UZ)
- [ ] Все API endpoints из docs/06 работают

## Промпт для Claude
[Заготовленный промпт из PROMPTS-LIBRARY.md]

## Зависимости
- Зависит от: DKR-100 (Auth модуль) ✅
- Блокирует: DKR-150 (Frontend каталога)

## Связанные документы
- docs/03-ARCHITECTURE.md → Products Module
- docs/04-DATABASE-SCHEMA.md → products
- docs/06-API-SPECIFICATION.md → Merchant Products

## Definition of Done
- [ ] Код написан и работает
- [ ] Тесты добавлены (coverage > 80%)
- [ ] PR создан и прошёл review
- [ ] PR merged в develop
- [ ] Smoke test на staging пройден
```

---

### Размерные классы задач

| Класс | Описание | Claude время | Review время | Пример |
|-------|----------|--------------|--------------|--------|
| **XS** | 1 файл, ~50 LOC | 5-10 мин | 5 мин | Добавить DTO |
| **S** | 1-2 файла, ~150 LOC | 15-30 мин | 10 мин | Один endpoint |
| **M** | 3-5 файлов, ~300 LOC | 30-60 мин | 20 мин | Модуль с CRUD |
| **L** | 5-10 файлов, ~600 LOC | 1-2 часа | 30 мин | Сложный модуль с интеграциями |
| **XL** | 10+ файлов, ~1000+ LOC | НЕ ДАВАТЬ | — | Разбить на S/M |

**Правило**: задачу класса L давать Claude только если нет возможности разбить.

---

### Пример декомпозиции большой фичи

❌ **ПЛОХО**: "Реализовать модуль Orders"

✅ **ХОРОШО**:
1. DKR-201: Prisma модели для Orders (XS)
2. DKR-202: Cart модуль базовый CRUD (M)
3. DKR-203: Pricing service (расчёт стоимости с НДС/скидками) (S)
4. DKR-204: Inventory reservation с Redis lock (M)
5. DKR-205: Orders.create() — создание заказа (M)
6. DKR-206: Orders.list / get / cancel (S)
7. DKR-207: Order status state machine (M)
8. DKR-208: Sub-orders для multi-merchant (M)
9. DKR-209: Event listeners (order.created, order.paid) (S)
10. DKR-210: E2E тест полного флоу заказа (S)

**Преимущества**:
- Каждая задача проверяема независимо
- Меньше merge conflicts
- Прогресс виден чаще
- Claude не теряется в большом скоупе

## Code review процесс

### Уровни review

#### Level 1: Автоматический (CI)
- ESLint
- TypeScript
- Unit tests
- E2E tests
- Test coverage
- Security scan (npm audit)

**Если упало — Claude фиксит до зелёного состояния**.

#### Level 2: Human review

**Tech Lead / Senior Dev смотрит**:

1. **Архитектура** (5 мин):
   - Правильный модуль?
   - Правильные слои?
   - Соответствует CLAUDE.md?

2. **Безопасность** (3 мин):
   - Guards на месте?
   - Multi-tenancy?
   - Sensitive data не утекает?

3. **Бизнес-логика** (5-10 мин):
   - Правильно ли реализовано?
   - Edge cases?
   - Тесты осмысленные?

4. **Производительность** (3 мин):
   - N+1?
   - Индексы?
   - Тяжёлые операции?

5. **Читаемость** (2 мин):
   - Понятный код?
   - Naming?

**Итого: 15-20 мин на PR класса M**.

---

### Шаблон review-комментариев

```markdown
## ✅ Что хорошо
- ...
- ...

## ⚠️ Что улучшить (non-blocking)
- ...
- ...

## ❌ Что блокирует (must fix)
- **<file>:<line>**: <issue>
  Предложение: <suggestion>

## Вопросы
- ...
```

### Когда Claude вносит правки

После комментариев — Claude вносит правки тем же PR за 15-30 мин:

```
Tech Lead в PR:
> Тут N+1 проблема. Используй include вместо .map с awaits.

Claude (в комментарии PR через 20 мин):
> Исправлено в коммите a1b2c3d. Заменил Promise.all с findUnique на 
> findMany с include. Tests проходят.
```

## Когда Claude не справится

### Сценарии, где нужен человек

#### 1. Архитектурные решения
- "Как структурировать модули?"
- "Monolith или микросервисы?"
- "Какую БД выбрать?"

**Решение**: Tech Lead принимает решение, документирует в ADR (Architecture Decision Record).

#### 2. Производственные инциденты
- Падение БД ночью
- Утечка памяти
- DDoS

**Решение**: Senior Dev / DevOps реагирует. Claude может помочь с анализом логов, но не с emergency action.

#### 3. Сложные дебаги
Особенно где причина не очевидна:
- Race conditions
- Memory leaks
- Сложные взаимодействия модулей

**Решение**: Senior Dev дебажит лично, Claude помогает с pattern matching.

#### 4. Переговоры и UX-решения
- Что лучше: модалка или новая страница?
- Какой текст ошибки понятнее?
- Сложные UX-флоу

**Решение**: Designer + PM решают, Claude реализует.

#### 5. Безопасность критичных модулей
- Платежи (особо)
- Auth и шифрование
- Финансовые расчёты

**Решение**: Senior Dev пишет лично или Claude пишет, но Senior Dev делает обстоятельный review.

#### 6. Performance оптимизация
- Профилирование запросов
- Анализ медленных endpoints
- Оптимизация SQL

**Решение**: Tech Lead профилирует, формулирует задачу для Claude.

## Стоимость разработки

### Затраты на Claude (Anthropic API)

**Допущения**:
- ~5-10 задач в день
- Каждая задача = ~50k input + ~30k output токенов
- Используется Claude Opus

**Расчёт**:
- 8 задач × 80k токенов × 22 рабочих дня = ~14M токенов в месяц
- Cost: $400-800 в месяц (зависит от подписки)

С учётом Claude Code как продукта (если используется через Anthropic UI):
- $200-400/месяц на разработчика
- 2-3 разработчика активно используют Claude
- = $400-1200 в месяц

**Итого на инструменты**: ~$1500/месяц.

## 💡 Best practices

### 1. Не торопись с Claude

Claude может писать код быстро, но **review должен быть тщательным**. Не мерджи "потому что зелёные тесты". Тесты могут быть неполными.

### 2. Версионируй промпты

Если промпт работает хорошо — сохрани в PROMPTS-LIBRARY.md. Если плохо — анализируй почему и улучшай.

### 3. Учи Claude на своём коде

Когда Claude делает ошибку и ты её исправляешь — добавь explicit правило в CLAUDE.md. Это работает как corporate memory.

### 4. Не давай Claude писать архитектуру

Архитектура — работа Tech Lead. Claude — реализация по архитектуре. Если Claude начинает "придумывать архитектуру" — останавливай.

### 5. Парное программирование с Claude

Для особо сложных задач — работай с Claude в реальном времени. Сложные алгоритмы, оптимизация запросов, дебаг — это не "выдать промпт и ждать", а интерактивно.

### 6. Документируй решения

В каждом сложном PR — Claude должен оставлять комментарий "Почему сделано так". Это сохраняет контекст для будущих разработчиков (включая будущего Claude).

### 7. Регулярная "проверка качества"

Раз в неделю — ревью кодовой базы. Что Claude писал хорошо, что плохо? Какие паттерны повторяются (хорошие/плохие)? Обновляй CLAUDE.md.

### 8. Не экономь на тестах

Тесты — это **главная страховка** от регрессий когда Claude меняет код. Без тестов = катастрофа.

### 9. Сохраняй "design intent"

Claude может изменить дизайн при рефакторинге. Если есть важные дизайнерские решения — комментируй в коде:
```typescript
// DESIGN: Используем именно этот алгоритм потому что X.
//         НЕ менять без обсуждения с Tech Lead.
```

### 10. Знай предел Claude

Не пытайся выжать из Claude то, что он не может. Признай — и сделай сам.

## 🎯 Финал: успешный AI-driven dev

Это **новый подход** к разработке. Не "Claude вместо разработчиков", а:

**Опытная человеческая команда + Claude = очень эффективный машина**

Tech Lead становится скорее **архитектором и тим-лидом**, чем "пишущим разработчиком". Claude — "стажёр-в-стероидах", который пишет код, но требует постоянного руководства.

При правильной организации проект «Домкрат» можно реализовать за **3-4 месяца с командой из 3-4 человек**, что было бы невозможно с традиционным подходом.
